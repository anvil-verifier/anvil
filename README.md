# Artifact Evaluation Instructions for "Compositional Verification of Cluster Control Planes"

This document is for SOSP'2026 artifact evaluation

## Artifact goal

The paper claims that Welder enables compositional verification of Kubernetes controllers, and that the verified controllers pass extensive testing and achieve competitive performance compared to unverified references. The key results to reproduce are:

1. **Verification results (§ 7.1 + Table 1).** Welder's framework, four controllers (VReplicaSet, VDeployment, VStatefulSet, VRabbitMQ) and their composition proof are fully verified by Verus (0 errors), with the code size table.
2. **Correctness results (§ 7.2 + Appendix Table 3).** The verified controllers pass (a) 629 functional tests generated by Acto, (b) 1302 controller-crash tests (868 individual + 434 correlated), and (c) 677 Pod-crash tests — with no bugs found.
3. **Performance results (§ 7.2 Table 2).** The verified controllers' `reconcile` and end-to-end reconciliation times are comparable to the unverified reference controllers (end-to-end differences within one standard deviation), and they do not add significant load on etcd or the API server.

We requires a Linux X86-64 machine to reproduce the result. You are welcomed to use Cloudlab machine (see 2.1) for reproduction.

## 3. Reproducing the correctness results

### 3.1 Functional tests (Acto)

Two test campaigns cover the four controllers. Both configs deploy **all four verified controllers together** (matching the paper's composition setting); the campaign determines which controller's custom resource is mutated:

```bash
cd ../acto
# Deployment + ReplicaSet campaign (~465 tests)
python3 -m acto --config data/vdeployment-controller/v0/config.json \
    --num-workers <N> --workdir testrun-vdeployment

# RabbitMQ + StatefulSet campaign (~164 tests)
python3 -m acto --config data/anvil-rabbitmq-controller/config.json \
    --num-workers <N> --workdir testrun-rabbitmq
```

Pick `<N>` based on your machine (each worker runs its own kind cluster; ~4 CPUs + 8 GB per worker is a reasonable rule of thumb). A standalone VReplicaSet config is also available at `data/vreplicaset-controller/v0/config.json`.

To inspect the results:

```bash
python3 -m acto.post_process.collect_test_result \
    --config data/vdeployment-controller/v0/config.json --testrun-dir testrun-vdeployment
```

This collects all test results into a CSV file under the testrun directory. Each row is one test step with the verdict of each oracle: `crash`, `health`, `consistency`, `operator_log`, `differential`, and `custom` (the per-controller oracles in `data/*/oracle.py`). The raw per-trial data (`mutated-*.yaml` desired states, `system-state-*.json` snapshots, `delta-*.log` diffs, and `generation-*-runtime.json` oracle outputs) lives in `testrun-*/trial-*/`; see ["Interpreting Acto's Test Result" in Acto's `docs/port.md`](https://github.com/xlab-uiuc/acto/blob/v-dev/docs/port.md) for a walkthrough of analyzing an alarm.

**Expected result:** no true alarms — every generated desired state is successfully reconciled (alarms, if any, should be inspected and are expected to be false alarms/misoperations, consistent with "No bug was found").

Keep the two testrun directories: they are the inputs for the crash tests (§3.2) and the performance measurement (§4).

### 3.2 Controller-crash and Pod-crash tests 

Chactos replays the desired-state sequences from a functional testrun while injecting pod-kill faults (ChaosMesh is installed automatically). The general form is:

```bash
python3 -m chactos --config <acto operator config> \
    --fi-config chactos/<fault injection config>.json \
    --testrun-dir <functional testrun dir> \
    --workdir <output dir> --num-workers <N>
```

Fault-injection configs, per test type in the paper's testing table:

| Test type | Campaign | `--fi-config` | `--config` / `--testrun-dir` |
|---|---|---|---|
| Individual controller crash | Deployment + ReplicaSet | `chactos/vdeployment.json`, `chactos/vreplicaset.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Individual controller crash | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller.json`, `chactos/vstatefulset.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |
| Correlated controller crash | Deployment + ReplicaSet | `chactos/vdeployment-correlated.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Correlated controller crash | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller-correlated.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |
| Pod crash (managed Pods) | Deployment + ReplicaSet | `chactos/vdeployment-pod-crash.json`, `chactos/vdeployment-pod-crash-rand.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Pod crash (managed Pods) | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller-pod-crash.json`, `chactos/rabbitmq-controller-pod-crash-rand.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |

How the configs map to the paper's test types:

* **Individual crash** configs select a single controller's pod by its `app.kubernetes.io/name` label.
* **Correlated crash** configs select `acto/tag: operator-pod`, which matches *all* deployed controller pods, so interacting controllers (e.g., Deployment + ReplicaSet) are crashed simultaneously.
* **Pod crash** configs select the *application* pods managed by the controllers (`app: pause-demo` for VDeployment, `app: test-cluster` for RabbitMQ). `pod_failure_ratio: -1` kills a uniformly random subset of the pods; the `-rand` variants additionally skip injection with probability 0.5 (`skip_fault_probability`) to diversify the schedules.

Each Chactos workdir contains per-trial directories mirroring Acto's layout; the oracle verdicts are in each trial's result files, and can be inspected the same way as in §3.1.

**Expected result:** in every trial the controllers recover from the crashes and the cluster converges to the desired state (no oracle violations). The test counts in the appendix table (552/276/401 for Deployment+ReplicaSet and 316/158/276 for RabbitMQ+StatefulSet) are the totals across these runs; crash tests skip trials whose input sequences are misoperations, so per-run counts can be slightly lower than the functional-test counts. These campaigns take hours to days depending on `--num-workers`.

## 4. Reproducing the performance results

The performance harness lives in `plugins/performance_measurement/`. It replays each functional test input against (a) the verified controllers ("anvil" phase) and (b) the unverified references ("reference" phase: `data/deployment-controller/v0/config.json` deploys the native Kubernetes Deployment/ReplicaSet controllers; `data/rabbitmq-operator/v2-19-2/config.json` deploys the official RabbitMQ cluster operator v2.19.2), while recording per-`reconcile` execution time, end-to-end reconciliation time (the "conditions" in `measure_runner.py`, including first-write latency), and control-plane resource usage (etcd/API server CPU, memory, and disk I/O via cAdvisor and the metrics API; a metrics-server is deployed from `data/metrics-server.yaml`).

**Step 1: generate the input workloads** from the functional testruns (§3.1):

```bash
cd ../acto
python3 plugins/performance_measurement/measure_performance.py --gen \
    --input-dir testrun-vdeployment \
    -c data/vdeployment-controller/v0/config.json -j vdeployment-controller

python3 plugins/performance_measurement/measure_performance.py --gen \
    --input-dir testrun-rabbitmq \
    -c data/anvil-rabbitmq-controller/config.json -j rabbitmq-operator
```

This writes `inputs/anvil_inputs/` and `inputs/reference/` under each testrun directory (the reference inputs are the same desired states converted to the reference controllers' CR format).

**Step 2: run the measurements.** The workdir names below are required — the result-processing script looks for exactly these directories:

```bash
python3 plugins/performance_measurement/measure_performance.py \
    --workdir testrun-vdeployment-performance-first-write \
    --input-dir testrun-vdeployment \
    -c data/vdeployment-controller/v0/config.json \
    -r data/deployment-controller/v0/config.json \
    -j vdeployment-controller

python3 plugins/performance_measurement/measure_performance.py \
    --workdir testrun-rabbitmq-performance-first-write \
    --input-dir testrun-rabbitmq \
    -c data/anvil-rabbitmq-controller/config.json \
    -r data/rabbitmq-operator/v2-19-2/config.json \
    -j rabbitmq-operator
```

Each run executes two modes per phase (`normal`: replay the input sequence; `single-operation`: apply each input from a fresh baseline). Use `--phase anvil` / `--phase reference` to run one side at a time.

**Step 3: process the results.** From the repository root (the parent of the two `testrun-*-performance-first-write` directories):

```bash
python3 plugins/performance_measurement/process_ts.py
```

This prints the latency comparison and control-plane resource statistics, and writes the summary table to `anvil-table-3.txt` (mean/max reconciliation latencies of the verified controllers vs. their unverified references). The raw measurements are kept under the two performance workdirs: `measurement_result_*.json` (per-input latency conditions, including the first-write latency) and `control_plane_stats/` (etcd and API-server CPU, memory, and disk I/O time series from cAdvisor and the metrics API).

**Expected result:** the absolute numbers depend on the platform, but you should observe the paper's key pattern — the verified controllers' `reconcile` times are modestly higher than the references (VRabbitMQ is faster than its reference), the end-to-end differences are negligible (within one standard deviation), and etcd/API-server CPU, memory, and disk I/O differences are within one standard deviation.

## Troubleshooting

* Acto/Chactos create kind clusters named per worker; if a run is interrupted, clean up with `kind get clusters | xargs -n1 kind delete cluster --name`.
* If image pulls from `ghcr.io` fail, authenticate to the GitHub container registry and retry, or preload images with `--images-archive`.
* Both Acto and Chactos are resumable at the campaign level: rerunning with the same `--workdir` starts fresh, so use distinct workdirs per attempt.
