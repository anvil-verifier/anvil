# Reproducing the Evaluation Results for "Welder: Compositional Verification of Cluster Control Planes"

This document describes how to reproduce the results in the Evaluation section of the Welder paper. It is written in the style of the [Anvil OSDI'24 artifact evaluation README](https://github.com/anvil-verifier/anvil/tree/osdi24), adapted to the Welder evaluation.

## Artifact goal

The paper claims that Welder enables pragmatic compositional verification of Kubernetes controllers, and that the verified controllers pass extensive testing and achieve competitive performance compared to unverified references. The key results to reproduce are:

1. **Verification results (§ Verification Effort).** Welder's framework, four controllers (VReplicaSet, VDeployment, VStatefulSet, VRabbitMQ) and their composition proof are fully verified by Verus (0 errors), with the code-size breakdown shown in the paper's code-size table.
2. **Correctness results (§ Controller Correctness, testing table in the appendix).** The verified controllers pass (a) 629 functional tests generated by Acto, (b) 1302 controller-crash tests (868 individual + 434 correlated), and (c) 677 Pod-crash tests — with no bugs found.
3. **Performance results (§ Controller Performance, performance table).** The verified controllers' `reconcile` and end-to-end reconciliation times are comparable to the unverified reference controllers (end-to-end differences within one standard deviation), and they do not add significant load on etcd or the API server.

Two repositories are involved:

| Repository | Branch | Used for |
|---|---|---|
| [`anvil`](https://github.com/anvil-verifier/anvil) | `main` | Result 1 (verification) — the verification scripts are the CI jobs in `.github/workflows/ci.yml` |
| [`acto`](https://github.com/xlab-uiuc/acto) | `v-dev` (this branch) | Results 2 and 3 (functional/crash testing via Acto + Chactos, performance measurement) |

The controller container images used by the tests are prebuilt and pinned in this repository (`ghcr.io/anvil-verifier/anvil/{vreplicaset,vdeployment,vstatefulset,rabbitmq}-controller:<commit-sha>` inside `data/*/deploy_remote.yaml`), so you do **not** need to build the controllers yourself to run the tests. To rebuild them from source, see `build.md` and `docker/controller/` in the `anvil` repository.

## 1. Reproducing the verification results

These steps mirror the `anvil` repository's CI (`main:.github/workflows/ci.yml`), which is the authoritative verification script.

**Environment:** Linux x86-64 (CI uses `ubuntu-24.04`). Verifying all controllers takes 7 minutes on a laptop with 8 cores and 16 GB memory.

**Step 1: setup toolchains**

```bash
git clone https://github.com/anvil-verifier/anvil.git && cd anvil

# Install Rust toolchain
curl --proto '=https' --tlsv1.2 -fsSL "https://sh.rustup.rs" | sh -s -- --default-toolchain none -y
. "$HOME/.cargo/env"
rustup toolchain install

# Download Verus release binary
curl -fsSL -o /tmp/verus.zip "https://github.com/verus-lang/verus/releases/download/release%2F0.2026.06.14.4ea7d0f/verus-0.2026.06.14.4ea7d0f-x86-linux.zip"
unzip -q /tmp/verus.zip -d "$HOME" && mv "$HOME/verus-x86-linux" "$HOME/verus"
export PATH="$PATH:$HOME/verus"
```

**Step 2: verification**

```bash
cargo verus verify --lib
```

This verifies the Anvil/Welder framework (including [TLA embedding library](https://github.com/anvil-verifier/verus-tla) as a dependency), all four controllers, and the composition proofs. **The expected result is 0 verification errors.** To verify individual pieces:

```bash
cargo verus focus --lib -- --verify-only-module vreplicaset_controller
cargo verus focus --lib -- --verify-only-module vdeployment_controller
cargo verus focus --lib -- --verify-only-module vstatefulset_controller
cargo verus focus --lib -- --verify-only-module rabbitmq_controller
cargo verus focus --lib -- --verify-only-module composition
```

### 1.1 Collecting the lines-of-code statistics

The scripts in `tools/` reproduce the code-size table in the paper (Trusted / Exec / Proof LoC per controller). They use Verus's `line_count` tool, which is **not** included in the prebuilt release binary from Step 1 — you additionally need a checkout of the Verus **source** to build `line_count` (only this small tool needs to be built, not Verus itself; the release binary from Step 1 is still used for `cargo verus`):

```bash
# Clone the Verus source (at the same version as the release binary) and build line_count
git clone --branch release/0.2026.06.14.4ea7d0f https://github.com/verus-lang/verus.git "$HOME/verus-source"
cd "$HOME/verus-source/source/tools/line_count" && cargo build --release

# Point the LOC script at the source checkout
export VERUS_DIR="$HOME/verus-source"
```

Then, from the root of this repository, run the push-button script:

```bash
./tools/gen-loc-table.sh
```

The script (1) compiles the library once with `--emit=dep-info` to enumerate all source files, (2) runs `line_count` over them to produce the combined per-file table `anvil_loc_table`, (3) categorizes the lines of each controller into `tools/<controller>-loc.json` via `tools/count-loc.py` (which classifies each file into trusted spec, trusted unverified, exec, model, conformance proof, guarantee proof, and CORE/ESR proof), and (4) prints the final code-size table via `tools/gen-t1.py`. The **expected output** looks like:

```
|                         |   Trusted Spec |   Trusted Unverified |   Exec | Conformance Proof   | Model   |   Core | ESR   |
|-------------------------|----------------|----------------------|--------|---------------------|---------|--------|-------|
| VReplicaSet controller  |            205 |                  257 |    300 | 119                 | 418     |   7726 | 7305  |
| VDeployment controller  |            254 |                  387 |    365 | 190                 | 563     |  10327 | 10201 |
| VStatefulSet controller |            194 |                  380 |    896 | 366                 | 1274    |  12085 | 11937 |
| RabbitMQ controller     |            247 |                  298 |   1619 | 175                 | 1660    |   6920 | 6772  |
| Composition proofs      |            181 |                    0 |      0 | --                  | 0       |    429 | --    |
| Total(all)              |           1081 |                 1322 |   3180 | 850                 | --      |  41402 | --    |
```

**Minor differences are expected** because the controllers' proofs have been optimized and updated to follow updates from upstream.

## 2. Environment setup for testing and performance (acto repository)

All remaining experiments run from the `acto` repository on branch `v-dev`. We suggest running the long campaigns inside `tmux`.

### 2.1 Option A: CloudLab (recommended)

The paper's experiments were run on CloudLab Clemson c6420 machines (dual Xeon Gold 6142, 384 GB DRAM, Ubuntu 22.04 LTS). We prepared a CloudLab profile that sets up the environment automatically. If you are new to CloudLab, see the [CloudLab documentation](https://docs.cloudlab.us/basic-concepts.html) and `docs/cloudlab.md` in the `acto` repository for account setup; if you do not already have a CloudLab account, apply for one at the [signup page](https://www.cloudlab.us/signup.php).

**Step 1: instantiate the profile.** Open the [`acto-cloudlab` profile](https://www.cloudlab.us/p/Sieve-Acto/acto-cloudlab?refspec=refs/heads/main) and keep hitting `Next` to create the experiment (pick physical node type `c6420` to match the paper's platform). CloudLab provisions the machine in ~5 minutes; wait until "Status" is `Ready` **and** "Startup" is `Finished` (the startup scripts install Docker, Go, kind, kubectl, and Python dependencies, and clone Acto under `~/workdir/acto`). If no machine is available, [submit a resource reservation](https://docs.cloudlab.us/reservations.html).

Note that by default you **only have access to the machine for 16 hours**; click the green `Extend` button to keep the machine long enough for the full campaigns.

**Step 2: switch to the evaluation branch and set up the environment.** Log into the machine (via `ssh` or the CloudLab web shell) and run:

```bash
cd ~/workdir/acto
git fetch origin v-dev && git checkout v-dev
./setup-welder-env.sh
```

### 2.2 Option B: your own machine

You need a Linux system with [Docker](https://docs.docker.com/engine/install/) and [Go](https://go.dev/doc/install) installed (any well-provisioned machine works, but absolute performance numbers depend on the platform). Then clone the repository and run the setup script:

```bash
git clone --branch v-dev https://github.com/xlab-uiuc/acto.git && cd acto
./setup-welder-env.sh
```

The script installs everything else without sudo: a Python 3.12 virtualenv at `./venv-welder` with all Python dependencies (provisioned via [uv](https://docs.astral.sh/uv/), so your system Python version does not matter), plus `kind` and `kubectl` under `~/.local/bin` if missing. The push-button script below picks up `./venv-welder` automatically.

### 2.3 Push-button script

All the commands in §3 and §4 are wrapped by `./reproduce-welder-testing.sh` in the `acto` repository root:

```bash
NUM_WORKERS=<N> ./reproduce-welder-testing.sh functional          # §3.1
NUM_WORKERS=<N> ./reproduce-welder-testing.sh crash               # §3.2 (requires functional results)
NUM_WORKERS=<N> ./reproduce-welder-testing.sh performance         # §4 (requires functional results)
NUM_WORKERS=<N> ./reproduce-welder-testing.sh all                 # everything, in order
```

The sections below explain what each phase does and the expected results; if you prefer running the individual commands manually, activate the virtualenv first (`source venv-welder/bin/activate`).

## 3. Reproducing the correctness results

### 3.1 Functional tests (Acto)

Two test campaigns cover the four controllers. Both configs deploy **all four verified controllers together** (matching the paper's composition setting); the campaign determines which controller's custom resource is mutated:

```bash
# Deployment + ReplicaSet campaign (~465 tests)
python3 -m acto --config data/vdeployment-controller/v0/config.json \
    --num-workers <N> --workdir testrun-vdeployment

# RabbitMQ + StatefulSet campaign (~164 tests)
python3 -m acto --config data/anvil-rabbitmq-controller/config.json \
    --num-workers <N> --workdir testrun-rabbitmq
```

Pick `<N>` based on your machine (each worker runs its own kind cluster; ~4 CPUs + 8 GB per worker is a reasonable rule of thumb). A standalone VReplicaSet config is also available at `data/vreplicaset-controller/v0/config.json`.

To inspect the results:

```bash
python3 -m acto.post_process.collect_test_result \
    --config data/vdeployment-controller/v0/config.json --testrun-dir testrun-vdeployment
```

This collects all test results into a CSV file under the testrun directory. Each row is one test step with the verdict of each oracle: `crash`, `health`, `consistency`, `operator_log`, `differential`, and `custom` (the per-controller oracles in `data/*/oracle.py`). The raw per-trial data (`mutated-*.yaml` desired states, `system-state-*.json` snapshots, `delta-*.log` diffs, and `generation-*-runtime.json` oracle outputs) lives in `testrun-*/trial-*/`; see ["Interpreting Acto's Test Result" in Acto's `docs/port.md`](https://github.com/xlab-uiuc/acto/blob/v-dev/docs/port.md) for a walkthrough of analyzing an alarm.

**Expected result:** no true alarms — every generated desired state is successfully reconciled (alarms, if any, should be inspected and are expected to be false alarms/misoperations, consistent with "No bug was found").

Keep the two testrun directories: they are the inputs for the crash tests (§3.2) and the performance measurement (§4).

### 3.2 Controller-crash and Pod-crash tests (Chactos)

Chactos replays the desired-state sequences from a functional testrun while injecting pod-kill faults (ChaosMesh is installed automatically). The general form is:

```bash
python3 -m chactos --config <acto operator config> \
    --fi-config chactos/<fault injection config>.json \
    --testrun-dir <functional testrun dir> \
    --workdir <output dir> --num-workers <N>
```

Fault-injection configs, per test type in the paper's testing table:

| Test type | Campaign | `--fi-config` | `--config` / `--testrun-dir` |
|---|---|---|---|
| Individual controller crash | Deployment + ReplicaSet | `chactos/vdeployment.json`, `chactos/vreplicaset.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Individual controller crash | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller.json`, `chactos/vstatefulset.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |
| Correlated controller crash | Deployment + ReplicaSet | `chactos/vdeployment-correlated.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Correlated controller crash | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller-correlated.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |
| Pod crash (managed Pods) | Deployment + ReplicaSet | `chactos/vdeployment-pod-crash.json`, `chactos/vdeployment-pod-crash-rand.json` | `data/vdeployment-controller/v0/config.json` / `testrun-vdeployment` |
| Pod crash (managed Pods) | RabbitMQ + StatefulSet | `chactos/rabbitmq-controller-pod-crash.json`, `chactos/rabbitmq-controller-pod-crash-rand.json` | `data/anvil-rabbitmq-controller/config.json` / `testrun-rabbitmq` |

How the configs map to the paper's test types:

* **Individual crash** configs select a single controller's pod by its `app.kubernetes.io/name` label.
* **Correlated crash** configs select `acto/tag: operator-pod`, which matches *all* deployed controller pods, so interacting controllers (e.g., Deployment + ReplicaSet) are crashed simultaneously.
* **Pod crash** configs select the *application* pods managed by the controllers (`app: pause-demo` for VDeployment, `app: test-cluster` for RabbitMQ). `pod_failure_ratio: -1` kills a uniformly random subset of the pods; the `-rand` variants additionally skip injection with probability 0.5 (`skip_fault_probability`) to diversify the schedules.

Each Chactos workdir contains per-trial directories mirroring Acto's layout; the oracle verdicts are in each trial's result files, and can be inspected the same way as in §3.1.

**Expected result:** in every trial the controllers recover from the crashes and the cluster converges to the desired state (no oracle violations). The test counts in the appendix table (552/276/401 for Deployment+ReplicaSet and 316/158/276 for RabbitMQ+StatefulSet) are the totals across these runs; crash tests skip trials whose input sequences are misoperations, so per-run counts can be slightly lower than the functional-test counts. These campaigns take hours to days depending on `--num-workers`.

## 4. Reproducing the performance results

The performance harness lives in `plugins/performance_measurement/`. It replays each functional test input against (a) the verified controllers ("anvil" phase) and (b) the unverified references ("reference" phase: `data/deployment-controller/v0/config.json` deploys the native Kubernetes Deployment/ReplicaSet controllers; `data/rabbitmq-operator/v2-19-2/config.json` deploys the official RabbitMQ cluster operator v2.19.2), while recording per-`reconcile` execution time, end-to-end reconciliation time (the "conditions" in `measure_runner.py`, including first-write latency), and control-plane resource usage (etcd/API server CPU, memory, and disk I/O via cAdvisor and the metrics API; a metrics-server is deployed from `data/metrics-server.yaml`).

**Step 1: generate the input workloads** from the functional testruns (§3.1):

```bash
python3 plugins/performance_measurement/measure_performance.py --gen \
    --input-dir testrun-vdeployment \
    -c data/vdeployment-controller/v0/config.json -j vdeployment-controller

python3 plugins/performance_measurement/measure_performance.py --gen \
    --input-dir testrun-rabbitmq \
    -c data/anvil-rabbitmq-controller/config.json -j rabbitmq-operator
```

This writes `inputs/anvil_inputs/` and `inputs/reference/` under each testrun directory (the reference inputs are the same desired states converted to the reference controllers' CR format).

**Step 2: run the measurements.** The workdir names below are required — the result-processing script looks for exactly these directories:

```bash
python3 plugins/performance_measurement/measure_performance.py \
    --workdir testrun-vdeployment-performance-first-write \
    --input-dir testrun-vdeployment \
    -c data/vdeployment-controller/v0/config.json \
    -r data/deployment-controller/v0/config.json \
    -j vdeployment-controller

python3 plugins/performance_measurement/measure_performance.py \
    --workdir testrun-rabbitmq-performance-first-write \
    --input-dir testrun-rabbitmq \
    -c data/anvil-rabbitmq-controller/config.json \
    -r data/rabbitmq-operator/v2-19-2/config.json \
    -j rabbitmq-operator
```

Each run executes two modes per phase (`normal`: replay the input sequence; `single-operation`: apply each input from a fresh baseline). Use `--phase anvil` / `--phase reference` to run one side at a time.

**Step 3: process the results.** From the repository root (the parent of the two `testrun-*-performance-first-write` directories):

```bash
python3 plugins/performance_measurement/process_ts.py
```

This prints the latency comparison and control-plane resource statistics, and writes the summary table to `anvil-table-3.txt` (mean/max reconciliation latencies of the verified controllers vs. their unverified references). The raw measurements are kept under the two performance workdirs: `measurement_result_*.json` (per-input latency conditions, including the first-write latency) and `control_plane_stats/` (etcd and API-server CPU, memory, and disk I/O time series from cAdvisor and the metrics API).

**Expected result:** the absolute numbers depend on the platform, but you should observe the paper's key pattern — the verified controllers' `reconcile` times are modestly higher than the references (VRabbitMQ is faster than its reference), the end-to-end differences are negligible (within one standard deviation), and etcd/API-server CPU, memory, and disk I/O differences are within one standard deviation.

## Troubleshooting

* Acto/Chactos create kind clusters named per worker; if a run is interrupted, clean up with `kind get clusters | xargs -n1 kind delete cluster --name`.
* If image pulls from `ghcr.io` fail, authenticate to the GitHub container registry and retry, or preload images with `--images-archive`.
* Both Acto and Chactos are resumable at the campaign level: rerunning with the same `--workdir` starts fresh, so use distinct workdirs per attempt.
